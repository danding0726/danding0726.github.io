> 这篇是 vLLM 部署的一份统一版实战笔记：先讲本地跑通，再讲生产落地要点，最后给出调优与容灾思路。

## 1）本地部署 vLLM：先把最小闭环跑通

先明确一个原则：**部署 vLLM 不是只看命令能不能跑，要同时看硬件与软件约束是否匹配。**

### 环境建议

- **Linux（推荐）**：GPU 场景最稳，安装链路最直接。  
- **Windows**：建议走 WSL + CUDA，注意 Docker/驱动兼容问题。  
- **macOS（Apple Silicon）**：可尝试，但 GPU 支持与编译链相对更复杂。

### 安装与启动（最小命令）

```bash
pip install vllm

python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen1.5-7B-Chat \
  --served-model-name qwen-7b-chat \
  --host 0.0.0.0 \
  --port 8000
```

### 服务验证

```bash
curl http://localhost:8000/v1/models
curl http://localhost:8000/health
```

如果返回正常，再接 Open WebUI 或你自己的网关层。

## 2）模型选择与显存：决定性能上限

一句话结论：**并发和吞吐瓶颈通常先出现在显存，而不是算力峰值。**

你需要同时考虑两部分开销：

1. **模型参数本体占用**（与参数规模、精度/量化位宽有关）
2. **KV Cache 占用**（与上下文长度、并发请求、batch 策略有关）

### 实践建议

- 优先做量化（INT8/INT4 等）评估，快速换取显存空间。
- 先做“目标显存预算”再选模型，不要反过来。
- 压测时必须包含真实上下文长度，否则会低估 KV Cache 压力。

## 3）快速上手 Demo：从 API 到 Web 交互

本地服务起来后，可以立刻做三件事：

- 用 `curl` 验证 API 可用性
- 用 Open WebUI 做交互测试
- 用你的业务 SDK 接 OpenAI-Compatible 接口

核心不是“能聊天”，而是先验证：

- 返回稳定性
- 首 token 延迟
- 长输出时的吞吐表现

## 4）生产部署四个关键点

## 4.1 容器化与 K8s 调度

生产推荐走 K8s：

- 使用 vLLM 官方镜像体系
- 配置资源请求/限制（CPU/GPU/内存）
- 通过滚动发布降低中断风险

## 4.2 可观测性（Observability）

至少接入：

- `/health` 健康探针
- `/metrics` 指标采集（Prometheus）
- Grafana 看板（QPS、队列长度、TTFT、tokens/s、错误率）

## 4.3 负载均衡与路由

生产里要避免“平均分流”的幻觉。建议结合：

- 请求长度感知
- 模型副本负载感知
- 超时与重试策略

![vLLM 负载与路由策略参考](/images/posts/2026-03-02-vllm-deploy/image2.png)

## 4.4 自动扩容与容灾

- 自动扩容：HPA 只是起点，GPU 场景建议叠加自定义指标。  
- 容灾：不仅依赖 K8s 自愈，还要有业务层兜底（路由降级、副本隔离、灰度切流）。

---

## 5）落地清单（团队可直接复用）

上线前至少确认这 6 项：

1. 目标模型在目标硬件上可稳定跑 24h
2. 压测覆盖真实上下文长度与峰值并发
3. 量化方案有 A/B 对比（质量 vs 成本）
4. 监控指标可追到“请求级”
5. 超时、重试、熔断策略已联调
6. 故障演练通过（单 Pod 故障 / 节点故障 / 突发流量）

---

## 6）总结

vLLM 的价值不是“把模型跑起来”，而是**把推理服务做成可运营的生产系统**：

- 本地阶段追求最小闭环
- 生产阶段追求稳定、可观测、可扩缩
- 运营阶段追求成本与质量的长期平衡

当这三层打通，vLLM 才真正发挥工程价值。
